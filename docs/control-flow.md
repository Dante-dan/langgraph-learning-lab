# 控制流：顺序、条件与循环

## 1. 顺序

固定边表示编译时已知的顺序：`START → check_weather → plan_route → END`。适合确定性工作流。

## 2. 条件

条件边的路由函数读取当前 State，返回下一个节点名或 `END`。路由函数应尽量保持纯净；网络调用、写库等副作用放在节点里。

确定性规则优先写代码。语义模糊时，模型节点可以用 structured output 生成受限枚举，例如 `walk | transit | clarify`，再由条件边读取。权限、额度、合规和危险动作仍须由代码复核。

## 3. 循环

条件边可以指回已经执行过的节点：

```text
generate → evaluate → accepted? → END
               └── no ─────────→ generate
```

可靠循环需要三层停止机制：

1. 业务成功条件，例如分数达到 0.8。
2. 业务尝试上限，例如最多修订 3 次。
3. `recursion_limit` / `recursionLimit` 作为运行时保险丝。

不要把数组遍历、固定重试和分页读取都画成图。普通 `for` 或重试库更清楚。只有每轮需要 checkpoint、动态路由、流式可见或人工介入时，图循环才值得使用。

## 4. 工具 Agent

工具 Agent 是一种动态循环：模型产生 `tool_calls`；条件边在 `tools` 与 `END` 之间选择；ToolNode 执行并写回结果；模型读取新观察后继续或结束。

每轮至少验证输入 schema、工具名、参数、权限、额度、超时、工具结果、最大步数和幂等性。
